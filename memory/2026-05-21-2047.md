# 2026-05-21 20:47 - v2.28: 删除卡顿彻底修复

## 问题
用户反馈点击删除键后需要2秒才会删除，且页面卡顿。

## 根本原因分析

### 直接原因：`localStorage.setItem(DB_KEY, JSON.stringify(db))` 阻塞UI

所有7个删除函数都在删除数据后**立即执行同步的 `localStorage.setItem(DB_KEY, JSON.stringify(db))`**，
其中包含 `JSON.stringify(db)` 序列化整个数据库对象（可能200KB+）。

在低性能电脑上，`JSON.stringify` 一个巨大嵌套对象耗时可达500ms-2s，阻塞了UI渲染。

### 深层原因：`forceSaveToFirebase` 也有同步开销

即使删除函数里的 `forceSaveToFirebase()` 不await（非阻塞），
JavaScript async 函数在遇到第一个 `await` 之前是同步执行的。

`_origSaveToFirebase` （即 `saveToFirebase`）在第一个 `await` 之前执行：
1. `repairData()` — 遍历所有prod日期数据（~60-90天×7车间）
2. `sanitizeForFirebase(db, 'saveToFirebase')` — 递归遍历整个数据库树

这两个函数都是同步且遍历整个数据库，进一步增加卡顿。

### 另一个问题：`forceSaveToFirebase` 中有冗余的 `sanitizeForFirebase`

`forceSaveToFirebase` 在调用 `_origSaveToFirebase` 前自己又执行了一次 `sanitizeForFirebase`，
但 `_origSaveToFirebase` 内部也做了一次，造成**重复的全量数据遍历**。

## 修复方案

### 1. 新增 `_deferredLocalStorageSave()` 辅助函数

```javascript
var _pendingLocalSave = false;
function _deferredLocalStorageSave() {
    if (_pendingLocalSave) return; // 同帧内多次触发只执行一次
    _pendingLocalSave = true;
    setTimeout(function() {
        _pendingLocalSave = false;
        try { localStorage.setItem(DB_KEY, JSON.stringify(db)); } catch(e) { ... }
    }, 0);
}
```

- 通过 `setTimeout(0)` 将 `JSON.stringify` + `localStorage.setItem` **延迟到下一个macrotask**
- 同帧内多次触发会合并为一次（`_pendingLocalSave` 防重复）
- 内存中的 db 先更新，渲染立即执行，浏览器UI不被阻塞

### 2. 所有7个删除函数同步更新

| 函数 | 文件 | 变更 |
|------|------|------|
| `delSysRecord` | 10-app-core.js | `localStorage.setItem` → `_deferredLocalStorageSave()` + `forceSaveToFirebase` → `setTimeout(→forceSaveToFirebase, 0)` |
| `delDLine` | 10-app-core.js | 同上 |
| `delProb` | 10-app-core.js | 同上 |
| `delLoss` | 10-app-core.js | 同上 |
| `delKaizen` | 10-app-core.js | 同上 |
| `delSimLine` | 10-app-core.js | 同上 |
| `equipDeleteRow` | 20-sqdip-adv.js | 同上（含fallback） |

### 3. `forceSaveToFirebase` 移除冗余 `sanitizeForFirebase`

`_origSaveToFirebase` 内部已经做 `sanitizeForFirebase`，外层再做是重复遍历。

## 执行路径对比

### 修复前（点击删除 → 2秒后才渲染）
```
用户点击删除 → 内存删除 → JSON.stringify(db) → localStorage.setItem → [500ms~2s阻塞]
→ forceSaveToFirebase: sanitizeForFirebase → repairData → sanitizeForFirebase → [更多阻塞]
→ render() 渲染 → UI更新
```

### 修复后（点击删除 → 瞬间渲染）
```
用户点击删除 → 内存删除 → _deferredLocalStorageSave(setTimeout(0)) → 立即返回
→ setTimeout(→forceSaveToFirebase, 0) → 立即返回
→ render() 渲染 → UI瞬间更新
                                      [下一macrotask]:
                                      → localStorage.setItem(JSON.stringify(db))
                                      → forceSaveToFirebase (以后台方式执行)
```

## 数据安全分析

- **localStorage写入延迟**：`setTimeout(0)` 通常4ms内执行，低端机<100ms
- **如果页面在延迟期间关闭**：下次加载从localStorage读取旧数据（含已删条目），然后Firebase同步修复
- **`forceSaveToFirebase`** 在下一macrotask异步执行，仍会写入Firebase和localStorage
- 双重保险：localStorage + Firebase，数据最终一致
